# Develop Diary — 开发日记整理

## 目标

将用户提供的原始日记内容，整理为与项目已有日记风格一致的 Markdown 文件，
写入 `开发日记/` 目录。整理过程包括：精简口语、提炼重点、分板块归类、
统一排版。

## 为什么这样做

开发日记是项目的活历史。散乱的流水账难以回顾，过度润色又失真。
此 skill 的目标是**在保留原意的前提下，让日记结构清晰、阅读高效**。

## 执行步骤

### 1. 读取已有日记，推断格式

先列出 `开发日记/` 目录下的所有 `.md` 文件，读取最近 1-2 篇，推断：
- **文件名格式**：通常是 `YYYY-MM-DD.md`
- **主标题格式**：通常是 `# YYYY-MM-DD Day N`
- **板块风格**：使用 `##` 作为板块标题，板块名由内容主题决定
- **Day 编号**：按已有最大编号递增；如果当天文件已存在，沿用其编号

### 2. 判断当日日记是否存在

检查是否存在 `开发日记/YYYY-MM-DD.md`（YYYY-MM-DD 为当天日期）。

- **如果已存在**：读取现有内容，将用户提供的新内容融入其中。
  已有板块补充或更新，新增内容创建新板块。不要完全覆盖。
- **如果不存在**：新建文件。Day 编号 = 已有最大 Day 编号 + 1。

### 3. 内容整理原则

对用户提供的原始内容进行以下处理：

| 处理项 | 说明 |
|--------|------|
| **精简** | 删除无意义的填充词（"然后然后"、"那个"），保留实质信息 |
| **分板块** | 按主题将内容归入不同 `##` 板块，如"开发进度"、"遇到的问题"、"明日计划"等 |
| **去口语化** | 将过于口语的表达改为简洁的书面语，但不变成八股文 |
| **保留个性** | 用户的情绪、吐槽、感慨可以适当保留，这是日记的灵魂 |
| **统一格式** | 列表、加粗、代码块等 Markdown 格式与已有日记保持一致 |

### 4. 输出文件

写入 `开发日记/YYYY-MM-DD.md`。

文件结构模板（根据实际内容调整）：

```markdown
# YYYY-MM-DD Day N

## 板块标题

内容...

## 另一个板块

内容...
```

### 5. 完成反馈

向用户简要汇报：
- 是新建还是更新
- 整理了哪些板块
- 文件路径

## 注意事项

- 不要擅自添加用户没有提到的内容，不要编造事实
- 如果用户提供的内容很少（几句话），也进行最小程度的整理后保存
- 如果用户明确说"覆盖"或"重写"，则完全替换当日日记；否则采用合并策略
- 保持中文标点规范，中英文之间加空格

# Ciff 扩展路径：50 → 5000+ 用户

## 全局视角

```
Phase 1: 50 → 200       加机器、调参数
Phase 2: 200 → 1000     应用层水平扩展，SSE 粘性会话
Phase 3: 1000 → 3000    数据库拆分，LLM Key 池化
Phase 4: 3000 → 5000+   微服务拆分，全分布式
```

```
Phase 1 (50→200)
  瓶颈：单机内存
  解法：调参数
  成本：0

Phase 2 (200→1000)
  瓶颈：单点故障
  解法：多实例 + 粘性会话
  成本：2-3 台服务器

Phase 3 (1000→3000)
  瓶颈：数据库 + LLM 限流
  解法：读写分离 + 本地模型 + BYOK
  成本：数据库从库 + Ollama GPU 服务器

Phase 4 (3000→5000+)
  瓶颈：团队协作 + 故障隔离
  解法：微服务拆分
  成本：K8s 集群 + 运维人力
```

## 一条始终不变的线

50 人 → 5000 人，**LLM API 限流始终是天花板**。架构再怎么演进，外部 API 的 RPM/TPM 限制不会消失。尽早投资本地模型（Ollama）+ BYOK，是最高性价比的扩展手段。

---

## Phase 1：50 → 200 人

### 触发条件

日均活跃超 80 人，峰值并发 SSE 连接超 30。

### 核心矛盾

单 JVM 512m 扛不住 200 个并发 SSE 长连接。

### 改什么

| 维度 | 变更 |
|------|------|
| JVM | -Xmx 调到 1g-2g |
| LLM | 多 API Key 轮询（同一供应商买多个 Key） |
| MySQL | max_connections 调到 100 |
| Redis | maxmemory 调到 256mb |
| PGVector | shared_buffers 调到 512M |
| 其他 | 日志级别调为 WARN，减少磁盘 IO |

### 不改什么

- Spring Boot 仍然单实例
- Docker Compose 部署
- 单库单表
- 不改代码，不改架构

### 改动量

改配置参数，一个人半天搞定。

---

## Phase 2：200 → 1000 人

### 触发条件

峰值 QPS 超 20，单实例 GC 频繁或宕机不可用超 5 分钟。

### 核心矛盾

单实例是单点故障，滚动更新和宕机都影响全部用户。

### 改什么

| 维度 | 变更 |
|------|------|
| 部署方式 | Docker Compose → 2-3 台服务器，Nginx upstream 多实例 |
| ciff-server | 部署 2-3 个实例，Nginx upstream 负载均衡 |
| SSE | Nginx ip_hash 粘性会话，同一用户请求打到同一实例 |
| LLM | API Key 池化（每供应商 3-5 个 Key 轮询） |
| MySQL | 连接池调到 30，加慢查询日志 |
| Redis | 部署 Redis Sentinel（1 主 2 从），保证高可用 |
| PGVector | 独立部署到单独服务器，给足内存 |
| 前端 | 构建产物上 CDN（或独立 Nginx 服务器） |

### 不改什么

- 单体 JAR 不拆
- SSE 实现逻辑不变
- Redis 仍然单分片
- 仍然 Docker 部署，不上 K8s
- 仍然走 Resilience4J

### Nginx 粘性会话配置

```nginx
upstream ciff_server {
    ip_hash;                        # same user → same instance
    server 10.0.0.1:8080;
    server 10.0.0.2:8080;
    server 10.0.0.3:8080;
}
```

### 改动量

部署脚本 + Nginx 配置 + Redis Sentinel，应用代码几乎不改。一个人 2-3 天。

## Phase 3：1000 → 3000 人

### 触发条件

MySQL 慢查询增多、对话表超百万行、单 Redis 内存不足、LLM 限流严重影响体验。

### 核心矛盾

数据库成为瓶颈，LLM API 限流无法通过加机器解决。

### 改什么

| 维度 | 变更 |
|------|------|
| MySQL | 读写分离（1 主 2 从），对话记录表按月分区 |
| Redis | 升级 Redis Cluster（3 主 3 从） |
| PGVector | 内存加大，ef_search 参数调优 |
| LLM | 本地模型扩容：部署 Ollama 集群（2-3 节点），分流 50% 请求 |
| LLM Key | 用户自带 Key（BYOK）：允许用户配置自己的 API Key |
| 异步任务 | Redis Stream → RabbitMQ，支持多消费者并发处理文档 |
| 监控 | Prometheus + Grafana 全链路监控 |
| 部署 | 引入 K8s（此时团队应该不止一个人了） |

### 不改什么

- 不分库分表
- Redis Cluster 对代码无感知
- @Async 保留
- Spring Actuator 保留
- 单体 JAR 仍然不拆

### LLM 限流应对策略

2000 用户 × 人均 5 条/天 = 10000 RPD。OpenAI Tier 2 ~5000 RPM 白天高峰可能够用，Tier 1 ~500 RPM 完全不够。

解法优先级：

1. **本地模型**（Ollama）承接非关键对话 → 降本 + 无限流
2. **用户自带 Key**（BYOK）→ 成本转嫁 + 无限流
3. **多 Key 池化** → 线性扩容
4. **请求排队 + 优先级** → 体验换容量

### 改动量

基础设施升级为主，应用代码少量适配（BYOK、RabbitMQ 消费者）。新基建需要 1-2 周。

## Phase 4：3000 → 5000+ 人

### 触发条件

团队 3+ 人协作开发，单体部署耦合严重，模块间互相影响。

### 核心矛盾

单体应用的团队协作效率，以及模块间故障隔离。

### 改什么

| 维度 | 变更 |
|------|------|
| 微服务拆分 | 按模块边界拆分（facade 接口天然就是服务契约） |
| ciff-chat | 独立服务，K8s HPA 自动扩缩容 |
| ciff-provider | 独立服务，LLM Gateway（Key 池 + 限流 + 路由） |
| ciff-knowledge | 独立服务 + PGVector 专属资源 |
| ciff-workflow | 独立服务 |
| 通信 | Facade 接口 → gRPC 或 Feign + MQ |
| MySQL | 分库：chat 独立库、agent 独立库 |
| PGVector | 评估迁移到 Milvus 集群 |
| 前端 | 微前端或独立部署 |

### 不改什么

- 业务逻辑不变
- 对话引擎逻辑不变
- Resilience4J 逻辑不变
- RAG 管道不变
- JSON 工作流引擎不变
- 表结构不变
- Vue 组件不变

### 拆分顺序

按故障隔离收益从高到低：

1. **先拆 ciff-provider**（LLM Gateway）→ 故障隔离最大收益
2. **再拆 ciff-knowledge**（独立资源）→ 数据库隔离
3. **ciff-chat 最后拆**（核心路径）→ 风险最高

每拆一个模块约 1 周，按优先级逐步推进。
